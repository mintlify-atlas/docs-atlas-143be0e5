TIMEPOINT Flash supports JWT-based authentication with Apple Sign-In for iOS apps. Authentication is optional and controlled by the `AUTH_ENABLED` environment variable.

## Authentication Modes

<Tabs>
  <Tab title="Open Access">
    **Default mode** - No authentication required.
    
    ```bash
    AUTH_ENABLED=false
    ```
    
    **Use cases:**
    - Local development
    - Internal tools
    - Demo environments
    - Self-hosted personal instances
    
    **Behavior:**
    - All endpoints accessible without tokens
    - No credit system enforcement
    - No user accounts
  </Tab>
  
  <Tab title="iOS App Mode">
    **Production mode** - Requires JWT Bearer tokens.
    
    ```bash
    AUTH_ENABLED=true
    JWT_SECRET_KEY=your_random_32_char_secret
    APPLE_BUNDLE_ID=com.yourcompany.timepoint
    SIGNUP_CREDITS=50
    ```
    
    **Use cases:**
    - iOS app deployments
    - Multi-tenant production
    - Public API access
    
    **Behavior:**
    - Protected endpoints require `Authorization: Bearer <token>`
    - Credit system enforced per user
    - Apple Sign-In integration
  </Tab>
</Tabs>

## Apple Sign-In Flow

### Architecture

```
iOS App                          TIMEPOINT Flash Backend
  │                                      │
  │  1. User taps "Sign in with Apple"  │
  │  ──────────────────────────────►     │
  │     ASAuthorizationAppleIDRequest    │
  │                                      │
  │  2. Apple returns identity_token     │
  │  ◄──────────────────────────────     │
  │     (RS256 JWT signed by Apple)      │
  │                                      │
  │  3. POST /api/v1/auth/apple          │
  │  ──────────────────────────────►     │
  │     { "identity_token": "..." }      │
  │                                      │
  │                                      │  4. Verify with Apple JWKS
  │                                      │     (RS256 signature check)
  │                                      │
  │                                      │  5. Find or create user
  │                                      │     (by apple_sub)
  │                                      │
  │                                      │  6. Grant signup credits
  │                                      │     (if first sign-in)
  │                                      │
  │  7. Return JWT pair                  │
  │  ◄──────────────────────────────     │
  │     { access_token, refresh_token }  │
  │                                      │
  │  8. Store refresh_token in Keychain │
  │     Keep access_token in memory      │
```

### Implementation Details

**Token verification** (`app/auth/apple.py`):

```python
def verify_apple_identity_token(identity_token: str) -> AppleTokenClaims:
    """
    Verifies Apple identity token using Apple's JWKS.
    
    - Algorithm: RS256
    - Issuer: https://appleid.apple.com
    - Audience: Your APPLE_BUNDLE_ID
    - Keys cached for 24 hours
    
    Returns: AppleTokenClaims(sub, email, email_verified)
    Raises: ValueError if invalid/expired
    """
```

**Key validation checks:**
1. **Signature verification** - RS256 with Apple's public keys
2. **Issuer check** - Must be `https://appleid.apple.com`
3. **Audience check** - Must match your `APPLE_BUNDLE_ID`
4. **Expiration check** - Token must not be expired
5. **JWKS caching** - Apple's signing keys cached for 24 hours

## JWT Token System

TIMEPOINT Flash uses a dual-token system: short-lived access tokens and long-lived refresh tokens.

### Access Tokens

**Format:** HS256 JWT

**Payload:**
```json
{
  "sub": "user_uuid",
  "iat": 1234567890,
  "exp": 1234568790,
  "type": "access"
}
```

**Lifetime:** 15 minutes (configurable via `JWT_ACCESS_EXPIRE_MINUTES`)

**Usage:**
```bash
GET /api/v1/timepoints
Authorization: Bearer <access_token>
```

**Generation** (`app/auth/jwt_handler.py`):
```python
def create_access_token(user_id: str) -> str:
    """Create a short-lived HS256 access token."""
    payload = {
        "sub": user_id,
        "iat": datetime.now(timezone.utc),
        "exp": now + timedelta(minutes=JWT_ACCESS_EXPIRE_MINUTES),
        "type": "access",
    }
    return jwt.encode(payload, JWT_SECRET_KEY, algorithm="HS256")
```

### Refresh Tokens

**Format:** Opaque random string (48 bytes, URL-safe base64)

**Storage:** SHA-256 hash stored in database (`refresh_tokens` table)

**Lifetime:** 30 days (configurable via `JWT_REFRESH_EXPIRE_DAYS`)

**Security features:**
1. **Hashed storage** - Only SHA-256 hash stored in DB
2. **Single use** - Each refresh invalidates the old token
3. **Rotation** - New token issued on every refresh
4. **Reuse detection** - Reusing revoked token triggers security lockdown

**Rotation flow:**
```python
async def rotate_refresh_token(
    session: AsyncSession, raw_old_token: str
) -> tuple[str, str]:
    """
    1. Hash old token and look up in DB
    2. Check if revoked (if yes, revoke ALL user tokens)
    3. Check if expired
    4. Revoke old token
    5. Generate and return new token pair
    
    Security: Reuse of revoked token → revoke all user tokens
    """
```

## Authentication Endpoints

### Apple Sign-In

<Card title="POST /api/v1/auth/apple" icon="apple">
  Exchange Apple identity token for JWT pair.
  
  **Request:**
  ```json
  {
    "identity_token": "eyJhbGciOiJSUzI1NiIs..."
  }
  ```
  
  **Response:**
  ```json
  {
    "access_token": "eyJhbGciOiJIUzI1NiIs...",
    "refresh_token": "dGhpc19pc19hX3JhbmRvbV90b2tlbg...",
    "token_type": "bearer",
    "expires_in": 900
  }
  ```
  
  **Behavior:**
  - First sign-in: Creates user + grants `SIGNUP_CREDITS`
  - Subsequent sign-ins: Returns new JWT pair
  - Updates `last_login_at` timestamp
</Card>

### Token Refresh

<Card title="POST /api/v1/auth/refresh" icon="rotate">
  Rotate refresh token and get new access token.
  
  **Request:**
  ```json
  {
    "refresh_token": "dGhpc19pc19hX3JhbmRvbV90b2tlbg..."
  }
  ```
  
  **Response:**
  ```json
  {
    "access_token": "eyJhbGciOiJIUzI1NiIs...",
    "refresh_token": "bmV3X3JhbmRvbV90b2tlbl9oZXJl...",
    "token_type": "bearer",
    "expires_in": 900
  }
  ```
  
  **Security:**
  - Old refresh token is revoked
  - Reusing old token triggers lockdown (all tokens revoked)
</Card>

### Current User

<Card title="GET /api/v1/auth/me" icon="user">
  Get authenticated user profile.
  
  **Headers:**
  ```
  Authorization: Bearer <access_token>
  ```
  
  **Response:**
  ```json
  {
    "id": "uuid",
    "email": "user@example.com",
    "display_name": "John Doe",
    "created_at": "2026-01-01T00:00:00Z",
    "last_login_at": "2026-03-06T12:00:00Z"
  }
  ```
</Card>

### Logout

<Card title="POST /api/v1/auth/logout" icon="right-from-bracket">
  Revoke refresh token (server-side logout).
  
  **Request:**
  ```json
  {
    "refresh_token": "dGhpc19pc19hX3JhbmRvbV90b2tlbg..."
  }
  ```
  
  **Response:**
  ```json
  {
    "detail": "Logged out"
  }
  ```
  
  **Note:** Always returns 200, does not leak whether token existed
</Card>

### Account Deletion

<Card title="DELETE /api/v1/auth/account" icon="trash">
  Soft-delete user account (required for App Store approval).
  
  **Headers:**
  ```
  Authorization: Bearer <access_token>
  ```
  
  **Response:**
  ```json
  {
    "detail": "Account deactivated"
  }
  ```
  
  **Effects:**
  - Sets `user.is_active = false`
  - Revokes all refresh tokens
  - Does NOT hard-delete data (preserves ledger integrity)
  - User can no longer authenticate
</Card>

## Dev Admin Endpoints

For testing authentication without an iOS device, use dev admin endpoints (gated behind `ADMIN_API_KEY`).

<Warning>
**Security:** Only enable admin endpoints in development/staging. Never set `ADMIN_API_KEY` in production.
</Warning>

### Create Test User

<Card title="POST /api/v1/auth/dev/token" icon="flask">
  Create test user and get JWT pair (bypass Apple Sign-In).
  
  **Headers:**
  ```
  X-Admin-Key: your_admin_key_here
  ```
  
  **Request:**
  ```json
  {
    "email": "test@example.com"
  }
  ```
  
  **Response:**
  ```json
  {
    "access_token": "eyJhbGciOiJIUzI1NiIs...",
    "refresh_token": "dGhpc19pc19hX3JhbmRvbV90b2tlbg...",
    "token_type": "bearer",
    "expires_in": 900,
    "user_id": "uuid"
  }
  ```
  
  **Behavior:**
  - Creates user with synthetic `apple_sub`
  - Grants `SIGNUP_CREDITS`
  - Returns JWT pair
  
  **Access control:**
  - `ADMIN_API_KEY` empty → 403 Forbidden
  - Wrong admin key → 403 Forbidden
</Card>

## Service-to-Service Authentication

For backend services calling TIMEPOINT Flash (e.g., billing microservice), use service key authentication.

### Configuration

```bash
FLASH_SERVICE_KEY=your_shared_secret_here
```

### Usage

<Card title="Service Key Header" icon="key">
  **Header:**
  ```
  X-Service-Key: your_shared_secret_here
  ```
  
  **Protected endpoints:**
  - `POST /api/v1/users/resolve` - Find or create user by `external_id`
  
  **Access control:**
  - `FLASH_SERVICE_KEY` empty → open access (no validation)
  - `FLASH_SERVICE_KEY` set → header must match
</Card>

**Example: User resolution**
```bash
POST /api/v1/users/resolve
X-Service-Key: your_shared_secret_here
Content-Type: application/json

{
  "external_id": "auth0|123456",
  "email": "user@example.com",
  "display_name": "John Doe"
}

# Response: User object (created or found)
```

## iOS Client Implementation Guide

### Token Storage

<Tabs>
  <Tab title="Refresh Token">
    **Storage:** iOS Keychain (`kSecClassGenericPassword`)
    
    **Rationale:**
    - Survives app restarts
    - Encrypted at rest
    - Secure from other apps
    
    **Implementation:**
    ```swift
    import Security
    
    func saveRefreshToken(_ token: String) {
        let data = token.data(using: .utf8)!
        let query: [String: Any] = [
            kSecClass as String: kSecClassGenericPassword,
            kSecAttrAccount as String: "timepoint_refresh_token",
            kSecValueData as String: data
        ]
        SecItemDelete(query as CFDictionary)
        SecItemAdd(query as CFDictionary, nil)
    }
    ```
  </Tab>
  
  <Tab title="Access Token">
    **Storage:** In-memory property (ObservableObject)
    
    **Rationale:**
    - Short-lived (15 minutes)
    - Re-derived from refresh on app launch
    - No need for persistent storage
    
    **Implementation:**
    ```swift
    @MainActor
    class AuthManager: ObservableObject {
        @Published var accessToken: String?
        
        func refreshAccessToken() async {
            guard let refreshToken = loadRefreshToken() else {
                // Redirect to sign-in
                return
            }
            
            // Call POST /auth/refresh
            let response = try await api.refresh(refreshToken)
            self.accessToken = response.access_token
            saveRefreshToken(response.refresh_token)
        }
    }
    ```
  </Tab>
</Tabs>

### App Launch Flow

```swift
struct TimepointApp: App {
    @StateObject private var auth = AuthManager()
    
    var body: some Scene {
        WindowGroup {
            if auth.isAuthenticated {
                MainView()
            } else {
                SignInView()
            }
        }
        .task {
            // On app launch, try to refresh token
            await auth.refreshIfNeeded()
        }
    }
}

class AuthManager: ObservableObject {
    @Published var isAuthenticated = false
    
    func refreshIfNeeded() async {
        guard let refreshToken = loadRefreshToken() else {
            // No refresh token, show sign-in
            return
        }
        
        do {
            let response = try await api.refresh(refreshToken)
            self.accessToken = response.access_token
            self.isAuthenticated = true
            saveRefreshToken(response.refresh_token)
        } catch {
            // Refresh failed, clear keychain and show sign-in
            clearRefreshToken()
            self.isAuthenticated = false
        }
    }
}
```

### Error Handling

<AccordionGroup>
  <Accordion title="401 Unauthorized - Token Expired">
    **Action:** Attempt token refresh, if that fails, redirect to Apple Sign-In
    
    ```swift
    func handleUnauthorized() async {
        do {
            await auth.refreshAccessToken()
            // Retry original request
        } catch {
            // Refresh failed, redirect to sign-in
            auth.logout()
        }
    }
    ```
  </Accordion>
  
  <Accordion title="403 Forbidden - Admin Key Invalid">
    **Cause:** Admin endpoint called without valid `X-Admin-Key`
    
    **Action:** This should only happen in testing. Check that `ADMIN_API_KEY` is set and matches your request header.
  </Accordion>
</AccordionGroup>

## Security Best Practices

<CardGroup cols={2}>
  <Card title="Generate Strong Secrets" icon="shield-halved">
    ```bash
    # JWT secret (32+ chars)
    python -c "import secrets; print(secrets.token_urlsafe(32))"
    
    # Service key
    python -c "import secrets; print(secrets.token_hex(32))"
    ```
  </Card>
  
  <Card title="Use Environment Variables" icon="lock">
    **Never commit secrets to git:**
    - Use `.env` files (gitignored)
    - Use secret management (Railway, Render, etc.)
    - Rotate keys periodically
  </Card>
  
  <Card title="Validate Bundle ID" icon="mobile">
    **Apple Sign-In validation:**
    - Set `APPLE_BUNDLE_ID` to your exact iOS bundle ID
    - Mismatch will reject all Apple tokens
    - Use different bundle IDs for dev/staging/prod
  </Card>
  
  <Card title="Disable Admin in Production" icon="triangle-exclamation">
    **Admin endpoints are dangerous:**
    - Only for dev/testing
    - Leave `ADMIN_API_KEY` empty in production
    - Use service keys for backend integrations
  </Card>
</CardGroup>
