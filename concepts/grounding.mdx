The **Grounding Agent** uses Google Search to verify historical facts before generation. This prevents hallucinations like placing the Deep Blue chess match in an "IBM server room" when it was actually held in a theater.

## Why Grounding?

Large language models can hallucinate plausible-sounding but incorrect details about historical events. Grounding solves this by:

1. **Searching Google** for verified facts about the event
2. **Extracting structured data** from search results
3. **Passing verified context** to downstream agents
4. **Preventing visual errors** by identifying physical presence

<Info>
Grounding uses Gemini's built-in Google Search integration, which provides real-time access to verified information with source citations.
</Info>

## When Grounding is Triggered

Grounding runs **only for historical queries** detected by the Judge agent:

```python
class GroundingInput:
    def needs_grounding(self) -> bool:
        """Grounding is triggered when query type is HISTORICAL."""
        return self.query_type == QueryType.HISTORICAL
```

<Note>
Generic period queries skip grounding because they don't require verification of specific facts. The LLM's training data is sufficient for plausible period details.
</Note>

## What Gets Verified

The Grounding agent produces a `GroundedContext` with 9 types of verified data:

```python
class GroundedContext(BaseModel):
    # Location verification
    verified_location: str       # "Equitable Center, 35th floor, Manhattan"
    venue_description: str       # "Theater-style room with raised seating"

    # Date verification
    verified_date: str           # "May 11, 1997"
    verified_year: int           # 1997

    # Participant verification
    verified_participants: list[str]  # ["Garry Kasparov", "Feng-hsiung Hsu", ...]

    # Physical presence (critical for image generation)
    physical_participants: list[str]  # ["Kasparov sitting at chess board", ...]
    entity_representations: list[str] # ["Deep Blue: IBM operator relaying moves"]

    # Event mechanics
    event_mechanics: str         # How the event physically worked
    visible_technology: str      # Period-accurate equipment descriptions
    photographic_reality: str    # What a photograph would actually show

    # Context
    historical_context: str      # Significance of the event
    source_citations: list[str]  # URLs from Google Search
    grounding_confidence: float  # 0.0-1.0
```

## Physical Presence Handling

The most critical grounding feature is **physical presence verification** — identifying WHO was literally visible in photographs.

### Non-Human Entities

Non-human entities (computers, AI, organizations) can't appear in images. The grounding agent discovers WHO represented them:

| Entity | Physical Representation |
|--------|------------------------|
| Deep Blue | IBM operator sitting across from Kasparov |
| "The Government" | The official who signed the document |
| HAL 9000 | Red camera lens on the wall |
| Soviet Union | Soviet officials in dark suits |
| "The Senate" | Senators seated in semicircle |

### Why This Matters for Images

Image generation models can't render abstract concepts. The grounding agent ensures:

<Check>Images show **people** where people were present</Check>
<Check>Technology is **period-accurate** (CRT monitors in 1997, not LCD)</Check>
<Check>Equipment is **positioned correctly** (separate room vs. on table)</Check>
<Check>Visual representations are **photographically realistic**</Check>

## Two-Step Grounding Process

Grounding uses a two-step process because Google Search and structured output can't be used simultaneously:

<Steps>
  <Step title="Step 1: Search & Retrieve">
    Call Gemini with Google Search grounding enabled to get verified facts as raw text:
    
    ```python
    grounded_response = await self.google_provider.call_text_grounded(
        prompt=full_prompt,
        model="gemini-2.5-flash",
        temperature=0.2,  # Low for factual accuracy
    )
    ```
    
    Returns: Unstructured text with search results and citations
  </Step>
  
  <Step title="Step 2: Parse to Structured Data">
    Parse the grounded text into structured `GroundedContext`:
    
    ```python
    parsed_response = await self.google_provider.call_text(
        prompt=PARSING_PROMPT.format(grounded_text=grounded_text),
        model="gemini-2.5-flash",
        response_model=GroundedContext,
        temperature=0.1,  # Very low for parsing accuracy
    )
    ```
    
    Returns: Structured `GroundedContext` object
  </Step>
</Steps>

## Error Handling

Grounding failures are **non-fatal** — the pipeline continues without verified facts:

```python
if result.success and result.content:
    state.grounded_context = result.content
    logger.info(f"Grounding: verified location='{result.content.verified_location}'")
else:
    # Grounding failure is non-fatal - log warning and continue
    logger.warning(f"Grounding failed: {result.error}")
    # Pipeline continues using LLM knowledge instead
```

**Location:** `app/core/pipeline.py:1005`

### Why Non-Fatal?

1. **Google Search might fail** — API errors, rate limits, no results
2. **Some events lack documentation** — Rare historical events may not have search results
3. **Text content still valuable** — User gets scene, characters, dialog even without verified facts

<Warning>
When grounding fails, image quality may suffer because physical presence can't be verified. The LLM may revert to common misconceptions.
</Warning>

## Downstream Usage

Grounded context flows to multiple agents:

```python
# Timeline agent uses verified dates
timeline_input = TimelineInput.from_judge_result(
    query, judge_result,
    grounded_context=state.grounded_context  # <-- Verified date/year
)

# Scene agent uses verified venue details
scene_input = SceneInput.from_timeline(
    query, timeline_data,
    grounded_context=state.grounded_context  # <-- Venue description, setting
)

# Characters agent uses verified participants
char_input = CharacterIdentificationInput.from_data(
    query, timeline, scene,
    grounded_context=state.grounded_context  # <-- Physical participants, entities
)

# Image prompt uses ALL grounded facts
image_prompt_input = ImagePromptInput(
    ...,
    grounded_context=state.grounded_context  # <-- Full verified context
)
```

## Example Prompt

The grounding prompt emphasizes physical presence:

```markdown
Research this historical event/person and provide VERIFIED facts:

QUERY: Deep Blue defeats Kasparov 1997
DETECTED FIGURES: Garry Kasparov
APPROXIMATE YEAR: 1997

Find and verify:
1. EXACT location (building name, floor, room type, city)
2. EXACT date (month, day, year)
3. WHO was actually there (named individuals, including operators, assistants)
4. WHAT the setting looked like (decor, layout, lighting)
5. Historical context and significance
6. HOW did this event physically work?
   - What was the physical setup/mechanics?
   - Who interacted with whom directly?
   - Where was any equipment/machinery located (same room or separate)?
7. WHAT technology or equipment was visible?
   - Describe appearance: size, color, style appropriate to the era
   - What would a photograph of this scene actually show?

=== CRITICAL: PHYSICAL PRESENCE FOR IMAGE GENERATION ===

8. WHO was PHYSICALLY VISIBLE in photographs of this event?
   - List each person WITH THEIR POSITION
   - Example: "IBM operator sitting across from Kasparov, making moves"

9. For any NON-HUMAN entities (computers, AI, organizations):
   - WHO was their HUMAN REPRESENTATIVE that would appear in photos?
   - Example: "Deep Blue" -> "IBM operator relaying moves"
   - HOW should this entity be visually depicted in an image?

Use Google Search to find accurate information.
Search for photographs and primary sources when possible.
Cite your sources.
```
